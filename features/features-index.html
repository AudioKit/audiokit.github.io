Why We Love AudioKit and You Might Too!

Make no mistake about it, we built AudioKit out of a deep love for sound and music, and huge amount of respect for people who make amazing audio apps. Our passion for creating the best possible environment for creating audio apps has led us to build a toolkit that excels on numerous fronts, each of which are discussed in more-depth from the links on the left. To boil it down to a few points, AudioKit was built to be:

  • Easy-to-learn and understand
  • The quickest way to develop a real-world audio application
  • Powerful enough to enable your most ambitious ideas
  • Evolving and improving over time

From these basic principles all of the features to left have followed.

Fast-development with Playgrounds

AudioKit 2.0 introduces the notion of a Playground, which is a little bit like Apple's Swift Playgrounds, but for Objective-C and much more powerful. AudioKit Playgrounds allow for rapid prototyping of an audio system, with instant feedback of how your code sounds. You no longer have to recompile your project after changing a single parameter to hear what the new value sounds like-just make your changes, and the Playground will already have updated itself. Once you've written and tested your audio system using Playgrounds, you can use that code in your main app.

We've developed a few Playgrounds for you to use, and are working on including many more!

Our vision for Playgrounds is to enable developers and sound-designers to be able to work quickly, efficiently, and more closely with each other during the audio-prototyping phase of app development.

Well-Named Classes and Parameters

We value clarity and beauty in code, which is why AudioKit utilizes well-named classes and parameters. Tradional Music-N style audio languages are characterized by succinctness at the expense of clarity. As a result, the code is often unclear and difficult to understand, especially as instruments grow more complex. The code below shows the creation and initialization of an FM oscillator in both Music-N and AudioKit; you be the judge.

Music-N
{% highlight bash %} asig foscili 1, 440, 1, 2, 3 {% endhighlight %}
AudioKit
{% highlight objective-c %} [[AKFMOscillator alloc] initWithWaveform:[AKTable standardSineWave] baseFrequency:akp(440) carrierMultiplier:akp(1) modulatingMultiplier:akp(2) modulationIndex:akp(3) amplitude:akp(1)]; {% endhighlight %}

Sensible Defaults

Sensible defaults allow you to create an AudioKit object without having to guess what initial values are appropriate—when you create an object, the parameter values will automatically be set to values that make sense. Of course, just as before, you can specify the value of all parameters in the init function, or use the defaults and change the values using properties.

As an example, many physical models sound best with specific parameters that you will always want, so for a canonical sleighbell sound, you can use the simplified version on the right:

Full InitializationSensible Defaults
{% highlight objective-c %} [[AKSleighbells alloc]initWithIntensity:akp(32) dampingFactor:akp(0.2) mainResonantFrequency:akp(2500) firstResonantFrequency:akp(5300) secondResonantFrequency:akp(6500)]; {% endhighlight %} Objective-C:
{% highlight objective-c %} [AKSleighbells sleighbells]; {% endhighlight %}
Swift:
{% highlight swift%} AKSleighbells(); {% endhighlight %}

Sensible Presets

In addition to sensible defaults, we give you a variety of default parameter sets lumped together into presets. For example, with the sleighbells above you could also create sleighbells wift softer bell sounds or an open bell sounds.

{% highlight objective-c %} [AKSleighbells presetSoftBells]; [AKSleighbells presetOpenBells]; {% endhighlight %}

Tight Xcode Integration

Write all your code in one place

High-level audio programming is often done in a complete separate environment from general application programming. With AudioKit, your audio programming is done in the same language that your application is written in. You only need one program and one language for both audio and app logic implementation.

Easy Installation

First and foremost, we strive to make AudioKit very easy to install. The examples should work out-of-the-box without any additional configuration. Configuration Tutorials and Walkthroughs are available on this site.

Clear Documentation

Complete Xcode documentation is accomplished with both in-line documentation, and through a downloadable docset. Xcode will provide on-the-fly pop-overs, code completion, and class information for all AudioKit objects, so you can easily access that information while you’re still in Xcode. In addition, we’ve made several file templates for both Objective-C and Swift.

AudioKit-specific File Templates

We’ve made AudioKit templates for creating instruments, processors, and conductor classes available in both Objective-C and Swift.

Many Operations

AudioKit has over 100 operations, including oscillators, physical models, reverbs, filters, and synthesizers. They are well-named and have named parameters, and we're working to make each operation fully-tested with sensible defaults. The operations are categorized into folders as follows:

CategoryDescription
Analysis Tracking classes that determine the amplitude and frequency of signals.
FFT Spectral functions including cross synthesis, mixing, scaling, warping, and resynthesis.
Mathematical Operations Basic operations such as scaling, mixing, summing, and multiplication of signals.
Signal Generators Classes for physical models, oscillators, linear controls, and granular synthesis textures.
Signal Input and Output Classes for handling incoming audio from devices and files as well as outputting audio to a speaker or files.
Signal Modifiers Effects such as convolution, delay, compression, flange, vibrato, reverb, panning, and many filters.

Powerful Sequences and Phrases

The sequencer in AudioKit is remarkable because it is not just limited to musical events. Music logic and app logic can appear side-by-side in events triggered by sequences. As a simple example, you can process the iPhone's sensor information and create audio events on the fly within a sequence. Alternatively, you can use musical events to trigger iOS functions like drawing graphics on the screen or even posting to Twitter.

When you don't need all that power, you can use the simpler syntax of phrases instead of sequences to play a series of notes on a given instrument.

High-level Utilities

AudioKit includes several built-in, high-level utilities that function as reusable components. These utilities will allow you to quickly add high-level features with a minimal amount of coding.

User Interface Tools

With AKTools, it is very easy to tie instrument properties to user interface elements such as sliders and buttons. You can also create your own custom user-interface utilities. We've provided the level-meter as an example of how to do this.

Instruments

Among these utilities are a sampler, pitch-tracker, and a level meter.

Additionally, there are several pre-made instruments included as well, and more will be added soon!

Plots

We've also included some components that allow for easy plotting of audio waveforms. These include:

  • Microphone input plotting
  • Audio output plotting
  • FFT plots
  • Instrument property plotting
  • Table plots
